# QSOE Rust Migration Plan

## Strategy

Move from evidence to implementation. The migration starts with observability,
fixtures, and host tools, then introduces Rust in small userland components
behind existing C boundaries. Only after that should deeper loader, libc,
task-manager, or kernel work be considered.

The expected path is:

1. Preserve the current bootable system.
2. Build a test and artifact-audit safety net.
3. Prove Rust can produce acceptable QSOE userland ELF artifacts.
4. Replace one small service.
5. Add reusable bindings.
6. Replace one resource-server driver.
7. Reassess deeper components with data.

## Phase 0: Baseline And Safety Net

Objective: make the current C system reproducible and measurable before any
Rust code enters the image.

Work:

- Capture exact release versions and component SHAs.
- Record host setup for macOS prebuilt runs and Debian/Ubuntu/container source
  builds.
- Add a `scripts/boot-smoke.sh` or equivalent wrapper around QEMU.
- Detect these boot milestones:
  - QSOE banner.
  - `/sbin/init` spawn.
  - `slogger` alive.
  - block device ready.
  - qrvfs mounted at `/usr`.
  - login prompt.
- Add a failure log bundle convention.
- Add a small artifact inspection script for existing C userland binaries.
- Document known current boot warnings, including seL4 untyped allocation
  messages, so future work does not confuse known noise with regressions.

Exit criteria:

- A clean release image boots to login under QEMU.
- The smoke script can distinguish success from timeout.
- Existing binaries have recorded ELF/relocation baselines.

## Phase 1: Host Tool Hardening

Objective: reduce image-building risk and create parser fixtures before touching
guest code.

Work:

- Add fixtures for GPT and qrvfs images generated by the current tools.
- Document expected image layout, file names, inode counts, and block counts.
- Port or wrap one host tool in Rust, preferably qrvfs inspection before qrvfs
  writing.
- Keep the existing tool as the oracle.
- Compare byte-for-byte output where deterministic output is expected.
- Compare semantic output where timestamps or ordering prevent byte equality.

Exit criteria:

- Rust host code can parse or inspect current release images.
- Fixture tests run on the host without QEMU.
- Current Python/C tools remain the default.

## Phase 2: Rust Toolchain Spike

Objective: prove Rust can build a minimal QSOE-compatible userland artifact.

Work:

- Add `rust/rust-toolchain.toml`.
- Add `rust/.cargo/config.toml`.
- Add a custom `riscv64-qsoe-user.json` target if the standard target is not
  suitable.
- Build a minimal `no_std`, `panic=abort` object.
- Link it through the same QSOE linker path used by `quser/common.mk`.
- Verify ET_EXEC, interpreter, dynamic tags, relocations, undefined symbols, and
  absence of TLS/unwind dependencies.
- Add artifact audit output to the build logs.

Exit criteria:

- A minimal Rust binary links with QSOE startup and libc.
- The artifact audit reports only supported relocation and runtime features.
- The binary is not installed by default.

## Phase 3: ABI And Resource-Server Bindings

Objective: define the Rust/C boundary once, then reuse it for pilots.

Work:

- Create `qsoe-abi` for constants and `#[repr(C)]` structures.
- Create `qsoe-ffi` for raw extern calls.
- Create `qsoe-ressrv` for safe resource-server wrappers.
- Add generated or manually checked layout assertions for all shared structs.
- Add C layout comparison tests where feasible.
- Write wrapper documentation for handle ownership, buffer lifetime, and message
  reply rules.

Exit criteria:

- Rust wrappers compile without pulling in `std`.
- Struct layout checks pass.
- Unsafe blocks are localized to FFI modules.

## Phase 4: First In-Guest Service: slogger-rs

Objective: replace one small service without changing external behavior.

Status: complete. `slogger-rs` passed opt-in and Rust-default RC evidence, then
the C `/sbin/slogger` service was retired in a separate removal PR.

Work:

- Specify current `slogger` behavior from `quser/sbin/slogger/main.c`.
- Add a focused in-guest smoke command if existing tests do not exercise
  `/dev/slog` enough.
- Implement `slogger-rs` using `qsoe-ffi` and `qsoe-ressrv`.
- Keep ring-buffer semantics equivalent.
- Add a narrow Rust selector for RC evidence.
- Build both C and Rust versions.
- Boot images with each version and compare logs.

Exit criteria:

- Rust `slogger-rs` is staged as `/sbin/slogger`.
- Rust `slogger-rs` boots to login and accepts log messages.
- Artifact audit passes.
- C rollback was available through the RC window and is now retired.

## Phase 5: Promote Resource-Server Patterns

Objective: turn lessons from `slogger-rs` into reusable service scaffolding.

Work:

- Refactor duplicated setup into `qsoe-ressrv`.
- Add examples for simple character-like and block-like resources.
- Document how resource names, channels, and request buffers are represented.
- Review error-code mapping from Rust to QSOE libc conventions.

Exit criteria:

- New Rust services can start with a small template.
- The template is tested with a trivial service.
- C services are unaffected.

## Phase 6: Driver Pilot: devb-virtio-rs

Objective: test Rust against real device state while preserving the external
block-device contract.

Status: complete. `devb-virtio-rs` passed opt-in and Rust-default file-read RC
evidence, then the C `/sbin/devb-virtio` driver was retired in a separate
removal PR.

Work:

- Specify current `devb-virtio` behavior and external file/device contract.
- Build typed wrappers for volatile MMIO.
- Build safe queue-state abstractions around unsafe descriptor memory.
- Keep all unsafe code in narrow modules.
- Add fixtures for virtio queue setup where host-side testing is possible.
- Boot with historical C driver and Rust driver images during the RC window.
- Run qrvfs mount and basic file access checks.

Exit criteria:

- Rust driver can expose `/dev/vblk0`.
- `/usr` mounts from qrvfs.
- Login still starts from on-disk init.
- Rust `devb-virtio-rs` is staged as `/sbin/devb-virtio`.
- C rollback was available through the RC window and is now retired.

## Phase 7: Shared Parsers And Format Libraries

Objective: move parser-heavy, testable logic into reusable Rust crates.

Candidates:

- qrvfs metadata.
- GPT partition parsing.
- CPIO archive parsing.
- syscfg and sysmap views.
- ELF read-only inspection.

Work:

- Add host fixtures from known-good release images.
- Keep libraries `no_std` compatible where reasonable.
- Use the host build for broad unit tests.
- Integrate into guest components only after host coverage is meaningful.

Exit criteria:

- Parsers reject malformed fixtures safely.
- Parser APIs do not expose raw buffer lifetimes unsafely.
- At least one parser is reused by both a host tool and a guest component.

## Phase 8: Additional Services And Drivers

Objective: migrate services with clearer boundaries before touching core
runtime.

Candidates:

- Small resource servers.
- Simple device servers.
- Focused admin utilities.
- Test helpers that exercise IPC and sync behavior.

Selection criteria:

- Small external surface.
- Existing or easy-to-add smoke tests.
- Minimal dependency on dynamic loader internals.
- Limited allocator requirements.
- Clear rollback path.

Exit criteria:

- At least two additional components run in Rust without expanding loader
  assumptions.
- Common wrappers have stabilized.

## Phase 9: Task Manager Internals

Objective: consider Rust only for internal modules that can be isolated from
process creation and relocation-critical paths.

Possible scope:

- Read-only parsers.
- Accounting tables.
- Bounded collections.
- Diagnostic formatting.
- Shared Rust archive packaging for multiple selected providers.

Deferred scope:

- Initial process spawn.
- Capability setup.
- ELF relocation application.
- Dynamic-linker interactions.

Exit criteria:

- A design review proves the module boundary does not affect boot-critical
  behavior.
- There is a direct rollback path.
- Multiple selected task-manager Rust providers link through one archive with
  one panic handler.
- Existing boot smoke and in-guest tests cover the affected path.

## Phase 10: Kernel Reassessment

Objective: decide whether Rust belongs in `nq` after userland evidence exists.

Inputs:

- Number of successful Rust userland components.
- Unsafe-code review outcomes.
- Toolchain reliability.
- Debuggability under QEMU.
- Artifact audit stability.
- Maintenance cost of mixed C/Rust builds.

Likely first kernel candidates:

- Pure data structure modules.
- Bounded parsers.
- Non-context-switch diagnostics.

Explicitly out of early scope:

- Trap entry.
- Context switching.
- Scheduler core.
- Boot assembly.
- seL4 interface assumptions.

## Release And Rollback Policy

- Every Rust migration ships behind a build flag until it has survived a release
  candidate period.
- The C implementation stays in tree until the Rust version has equivalent boot,
  smoke, and artifact-audit evidence.
- If a Rust component fails boot smoke, the default must automatically remain or
  return to the C implementation.
- Release notes must list every component whose implementation language changed.
- Use `RELEASE_NOTE_TEMPLATE.md` for any Rust opt-in selector, Rust-default
  release candidate, or C retirement note.
